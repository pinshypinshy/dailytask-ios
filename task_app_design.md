# タスク管理アプリ 設計書（MVP）

## 0. 本書の前提

本設計書は以下の前提に基づく。前提が変われば設計も変わるため、最初に明示する。

- **プラットフォーム**: iOS / SwiftUI + SwiftData
- **タスクの性質**: その日限りの単発型を基本とする。毎日繰り返す習慣型は本MVPの対象外（拡張機能として後述）。
- **データ構造**: 内部正本は SwiftData。テキストファイルはエクスポート／インポート用の表現（二層構造）。文法は独自文法ではなく JSON ベースのスキーマ検証。
- **タスクとタグの関係**: 多対多（1タスクが複数タグに属せる）。

---

## 1. データモデル

### 1.1 エンティティ一覧

| エンティティ  | 役割                                         |
| ------------- | -------------------------------------------- |
| Task          | 1日分のタスク1件                             |
| Tag           | タスクを分類するタグ                         |
| DailySnapshot | 日次の達成状況を確定保存する記録             |
| BackupFile    | エクスポート／インポート用ファイルのメタ情報 |

### 1.2 Task

| フィールド  | 型     | 説明                                           |
| ----------- | ------ | ---------------------------------------------- |
| id          | UUID   | 一意識別子                                     |
| name        | String | タスク名                                       |
| isFavorite  | Bool   | お気に入りフラグ                               |
| isCompleted | Bool   | 実行済みフラグ                                 |
| date        | Date   | このタスクが属する日（日付のみ、時刻は正規化） |
| tagIDs      | [UUID] | 所属タグのID配列（多対多をタスク側で保持）     |
| createdAt   | Date   | 作成日時                                       |

補足：当初の設計ではタグ側だけが「含まれるタスク」を持っていたが、タスク→タグの逆引きができず非効率なため、タスク側に `tagIDs` を持たせて双方向に解決できるようにした。

### 1.3 Tag

| フィールド | 型     | 説明                                                 |
| ---------- | ------ | ---------------------------------------------------- |
| id         | UUID   | 一意識別子                                           |
| name       | String | タグ名                                               |
| colorHex   | String | カラー（16進文字列で保持。後述のプリセットから選択） |

補足：タグが「含まれるタスク」を直接保持する設計はやめ、Taskの `tagIDs` から導出する。冗長な相互参照による不整合を防ぐため。

### 1.4 DailySnapshot（当初設計に無かった追加エンティティ）

達成率の分母が後から改変されるのを防ぐための記録。日付が変わるタイミング、またはその日のタスク編集時に確定・更新する。

| フィールド     | 型                                   | 説明                       |
| -------------- | ------------------------------------ | -------------------------- |
| date           | Date                                 | 対象日                     |
| totalCount     | Int                                  | その日のタスク総数（分母） |
| completedCount | Int                                  | 実行済みタスク数（分子）   |
| perTag         | [UUID: (total: Int, completed: Int)] | タグ別の分母・分子         |

補足：達成率をリアルタイム計算すると、過去のタスクを削除した際に過去の達成率まで変わってしまう。これを防ぐため、日次でスナップショットを確定保存し、グラフはスナップショットを参照する。当日分のみリアルタイム再計算する。

### 1.5 BackupFile

| フィールド   | 型     | 説明                           |
| ------------ | ------ | ------------------------------ |
| id           | UUID   | 一意識別子                     |
| fileName     | String | 表示名                         |
| isActive     | Bool   | アプリ表示に反映中のファイルか |
| lastModified | Date   | 最終更新日時                   |

## 2. 画面仕様（TabView 3構成）

### 2.1 今日のタスク

- 当日の Task 一覧を表示する。
- 各タスク行：タスク名、お気に入りトグル、実行済みチェックボックス。
- 操作：タスクの追加・編集・削除。タグの追加・編集・削除。
- **お気に入りの用途を定義**：お気に入りタスクを一覧上部にピン留め表示する。当初設計では用途が未定義だったため、並び替え基準として機能を与えた。
- **タグ削除時の挙動**：タグを削除しても Task 自体は残し、各 Task の `tagIDs` から該当IDを除去するのみとする。過去の DailySnapshot の `perTag` 記録は保持する（過去のグラフを壊さないため）。
- **空状態**：タスク0件のときは「今日のタスクはありません」を表示。

### 2.2 達成状況グラフ

- 折れ線グラフ。横軸＝時間、縦軸＝達成率（0〜100%）。
- データソースは DailySnapshot（当日のみリアルタイム算出）。
- **横軸の粒度**：週表示は1日単位、月表示は1日単位（点が多い場合は間引き表示も検討）。
- 期間切り替え：デフォルトは今週。左右矢印で前週・次週へ移動。今月表示にも切り替え可能。
- タグフィルタ：タグを複数選択すると、選択タグに属するタスクの達成率に切り替わる。
  - **集計方法を定義**：選択タグの**和集合**とする。複数タグに属するタスクは1回だけ計上し、二重計上しない（分母は和集合のユニークタスク数）。
  - 当初設計では和集合／積集合および二重計上の扱いが未定義だったため、和集合・重複排除と定義した。
- **空状態（0/0）**：その日にタスクが存在しない場合は達成率を「データなし」として点を打たない（0%と区別する）。

### 2.3 保存データ

- エクスポート／インポート用ファイルの一覧。
- デフォルトで1ファイルが存在し、内部データ更新時に同期書き出しする。
- ファイルの追加が可能。複数ある場合、アプリ表示に反映する `isActive` なファイルをユーザーが1つ選択する。
- ユーザーがファイルを編集して保存する際、スキーマ検証を行う。不正なら**エラー表示して保存を拒否**する。
- ファイル追加時も同様に検証し、不正なら追加を拒否する。
- **二層構造の方針**：正本は SwiftData。ファイルは「読み込み時に検証→内部DBへ反映」「内部更新時に書き出し」という橋渡し役とし、ファイル自体を正本にしない（同期バグと文法バグによる全損を避けるため）。

## 3. 保存ファイルのフォーマットと検証

### 3.1 フォーマット

独自文法ではなく JSON を採用する（パーサ・バリデータが既製で使え、保守コストが低い）。スキーマ例：

```json
{
  "version": 1,
  "tags": [{ "id": "uuid", "name": "勉強", "colorHex": "#4A90D9" }],
  "tasks": [
    {
      "id": "uuid",
      "name": "数学の課題",
      "isFavorite": false,
      "isCompleted": true,
      "date": "2026-06-15",
      "tagIDs": ["uuid"]
    }
  ]
}
```

### 3.2 検証ルール（保存・追加時に適用）

1. JSON としてパース可能か。
2. `version` が既知のバージョンか。
3. 必須フィールドの存在と型の一致。
4. `date` が ISO 8601（日付）形式か。
5. `colorHex` が許可された形式・プリセットか。
6. `tagIDs` が実在する `tags[].id` を参照しているか（孤立参照の検出）。
7. `id` の重複がないか。

いずれか1つでも違反すれば、どのルールに違反したか（行・フィールド名）を含むエラーメッセージを表示し、保存／追加を中止する。

## 4. 主要ロジック

### 4.1 達成率算出

- 当日：`completedCount / totalCount`（0件なら「データなし」）。
- 過去：DailySnapshot を参照（再計算しない）。
- タグフィルタ時：選択タグの和集合に属するユニークタスクで分母・分子を算出。

### 4.2 日付繰り越し

- 日付が変わると、前日分は DailySnapshot として確定済みになる。
- 新しい日は空のタスクリストから始まる（単発型のため、前日のタスクは繰り越さない）。

### 4.3 ファイル同期

- 内部DB更新 → `isActive` なファイルへ書き出し。
- ファイル編集保存 → 検証 → 通過時のみ内部DBへ反映。

---

## 5. MVP対象外（拡張候補）

- 繰り返し・習慣型タスク（テンプレート＋繰り返しルール）。
- 月表示でのデータ点間引き・集約表示。
- タグカラーの自由選択（MVPはプリセット固定）。
- 複数ファイル間の差分マージ。
